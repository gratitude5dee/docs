Zap separates recipe planning from provider execution. When a run is submitted, the provider router inspects each step's `provider` field (or the recipe's `defaults.provider` fallback) and selects the matching adapter. This decoupling means the same recipe can be planned for free (dry-run priced against the model rate table) and later dispatched live against any of the four supported BYOK providers without changing a single step definition.

<Note>
  **Zap v0.2 removed the `mock` provider.** The only valid values for `provider` and `defaults.provider` are `gmi`, `fal`, `prodia`, and `runware`. For zero-spend development, use **plan mode** — `zap run` without `--live` — which validates the recipe, expands `video.extend` chains, and prints a priced plan without dispatching any provider jobs.
</Note>

## Supported Providers

Zap v0.2 recognises four production provider adapters. All are bring-your-own-key (BYOK) — Zap does not proxy or resell provider capacity.

### `gmi`

The **default** live provider for image and video. Routes to [GMI Cloud](https://console.gmicloud.ai) and settles on the flagship Seedance 2.0 (`seedance-2-0-260128`) model for video. Requires a `gmi_api_key` (and optionally `gmi_org_id`). This is the value used when `defaults.provider` is omitted.

For the full BYOK setup, request/response shapes, T2V / I2V / R2V modes, and resolution pricing, see [GMI Cloud (BYOK)](/providers/gmi).

### `fal`

Secondary live provider for image, video, and audio via the fal.ai queue. Requires a `fal_key`. Common defaults: `fal-ai/flux/dev` for `image.gen`, `fal-ai/kling-video/v2.1/pro/image-to-video` for `video.gen`.

### `prodia`

Async image generation through Prodia's `/v2/job` API. Requires a `prodia_token`.

### `runware`

Async image generation with Runware's `getResponse` polling API. Requires a `runware_key`.

## Plan Mode vs Live Mode

`zap run` defaults to **plan mode**: schema validation, `video.extend` expansion, per-step USD estimation against the built-in rate table, and budget enforcement — but **no provider calls**. The result is stored under `.zap/runs/<runId>/result.json` with `mode: "plan"` and `status: "planned"`.

```bash
# Zero-spend plan
npx @wzrdtech/zap@0.2.0 run agent/skills/zap-my-zap/Zap.md --json

# Same command, dispatched live
npx @wzrdtech/zap@0.2.0 run agent/skills/zap-my-zap/Zap.md --live --input PROMPT="…"
```

Live runs require both a saved BYOK key for the target provider and all `required: true` inputs. Plan mode fills missing required inputs with deterministic placeholders (`mock-<name>`) purely for cost estimation.

## Storing Keys — CLI

The CLI keeps a local encrypted credential store under `~/.zap/credentials.json`. Manage entries with `zap keys`:

```bash
# Save a key (last 4 characters are printed on success)
zap keys add gmi gmi_api_key --value "$GMI_API_KEY"
zap keys add fal fal_key --value "$FAL_KEY"

# List saved keys (values are never printed)
zap keys list

# Round-trip check a key against the provider
zap keys test fal

# Remove a key
zap keys remove fal_key

# Push local keys to the hosted Supabase vault (requires `zap login`)
zap keys sync
```

## Storing Keys — Web

For runs initiated from `zap.wzrd.tech`, keys are stored server-side per creator wallet through the `zap-user-secrets` Supabase Edge Function. Keys are encrypted at rest and only decrypted immediately before dispatching a step. See [Supabase Secrets Setup](/deployment/supabase-secrets) and [Auth & Secrets](/webapp/auth-secrets).

## BYOK Secret Types

The following secret types are recognised by both the CLI credential store and the Supabase vault:

| Secret Key | Provider | Description |
| --- | --- | --- |
| `gmi_api_key` | GMI Cloud | Primary API key for GMI image and video generation |
| `gmi_org_id` | GMI Cloud | Optional organisation ID scoping requests to a GMI workspace |
| `fal_key` | fal.ai | API key for fal.ai model inference |
| `runware_key` | Runware | API key for Runware image generation |
| `prodia_token` | Prodia | API token for Prodia image generation |
| `openrouter_key` | OpenRouter | API key for OpenRouter LLM and multimodal routing (used by the optional VLM judge) |
| `ai_gateway_api_key` | AI Gateway | API key for a self-hosted or cloud AI Gateway instance (used by the optional VLM judge) |

<Note>
  Provider adapters return **run-safe metadata only**: step IDs, status strings, output URLs, observed cost, and error messages. They never return large media blobs to the agent context. Media assets are stored at the URL returned by the adapter and fetched lazily by the web UI or downstream steps.
</Note>

## Judge Scoring

Steps may declare an optional `judge` block to request an automated VLM quality score after the step completes. Judge scoring routes through the AI Gateway (`ai_gateway_api_key`) with an optional model override via the `ZAP_JUDGE_MODEL` environment variable. When no gateway key is configured the runtime records a deterministic `heuristic` feedback score instead of a VLM label.

<Warning>
  Never commit provider API keys to source control. Do not hardcode `gmi_api_key`, `fal_key`, or any other secret in `Zap.md`, prompt files, or `SKILL.md`. Store all keys via `zap keys add` (CLI) or the Supabase vault (web), and rotate them immediately if accidentally exposed. See [Supabase Secrets Setup](/deployment/supabase-secrets) for the recommended key storage pattern.
</Warning>
